Measure, don’t guess!

任何需要量化的性能，都需要数据来支撑，没有数据的性能比较，只能算猜测。JMH——Java Micro Benchmark 是使用java语言开发的基准测试框架，可以做精确到微妙级别的性能测试，由OpenJDK/Oracle 里面那些开发了 Java 编译器的大牛们所写，质量绝对保证~

3 JMH概念解释

3.1 预热

JMH中是用@Warmup来对预热进行设置，因为JVM的相关机制(JIT)，如果某个函数被调用多次之后，JVM 就会尝试将其编译成为机器码从而提高执行速度。所以为了让 benchmark 的结果更加接近真实情况就需要进行预热。至于预热的次数，JMH samples中来看，用到的地方均和正式测试@Measurement的配置一样

3.2 测量维度

测量的维度主要通过枚举类Mode来体现，有4个取值

  • Throughput: 整体吞吐量. 单位时间内能执行的次数

  • AverageTime: 调用的平均时间，每调用一次所花时间

  • SampleTime: 随机取样，最后输出取样结果的分布，例如“99%的调用在xxx毫秒以内，99.99%的调用在xxx毫秒以内”

  • SingleShotTime: 以上模式都是默认一次 iteration 是 1s，唯有 SingleShotTime 是只运行一次。往往同时把 warmup 次数设为0，用于测试冷启动时的性能。

最常用的是前两种模式 Throughput, AverageTime

3.2 常用注解解释

@Benchmark

方法注解，表示该方法是一个基准测试方法，类比Junit的@Test注解

@State

用来声明状态，

可以用在类上表示该类所有方法均生效，或者用在方法上，表示只对该方法生效。
其枚举类型Scope取值有三种：

  • Benchmark

    这个表示的是，在基准测试阶段，所有的线程共享同一份@Benchmark测试方法的拷贝.可以理解被基准测试的方法是一个单例，其中一个线程如果在调用该方法，另外的线程需要等待调用结束才能调用，而不是每个线程各管一份测试方法，该模式与Thread对立

  • Thread

    这个表示的是，在基准测试阶段，所有的线程各自占有一份@Benchmark测试方法的拷贝.可以理解成单例的对立面

  • Group

    这个表示的是，在基准测试阶段，一个组的线程占用一个@Benchmark测试方法

@BenchmarkMode

枚举类Mode有5种取值，除了 < 2.1.2 测量维度 > 中介绍的4种取值以外，还有一个All的取值，表示4种模式全都测试一遍

@OutputTimeUnit

最终显示测试结果时，使用的时间单位，设置适合的结果展示单位，可以一目了然看出测试结果

@Warmup

可以用在类上表示所有类中测试方法的预热设置，或者用在方法上表示该方法的预热设置

示例设置：@Warmup(iterations = 3, time = 1, timeUnit = TimeUnit.SECONDS)

iterations：迭代次数 设置为 3

time + timeUnit : 1 + TimeUnit.SECONDS = 1 秒钟， 即时间单元为 1 秒钟。

连起来表示： 在 1 秒钟内反复调用@Benchmark测试方法，持续 5 个时间单元。

举例来说，demo1里面的测试方法：

该方法执行一次大约花费 0.5 秒，所以在一个时间单元(1秒钟)内，大约调用 2 次，这也就是吞吐量的数据来源。

那么引申出一个疑问，假设测试方法执行一次最少需要花费2秒钟，但是测试的时间单元设置的是1秒钟，那会怎么样呢？答案是JMH会保证至少调用一次测试方法。也即若时间单元小于测试方法执行一次的时间，那么就执行一次测试方法，为了数据准确性，尽量不要这样设置。

@Measurement

用法基本上同 @Warmup，可设置的参数也和@Warmup一样，表示的是正式测试的设置

@Fork

可以设置Fork进程数，设置为几，就表示启动几个进程来测试

平常测试一个方法花费的时间，我们通常会用一个for循环来测试，然后循环后的时间减去循环前的时间再除以循环次数，来评估出执行一次的时间，但是这样的测试是会有比较大的误差的。看下例：

循环次数 第1次执行时间 第2次执行时间 第3次执行时间 平均时间
10万次 866 991 978 945 毫秒
20万次 1037 1010 1055 1034 毫秒

上述结果很明显了，如果10万次的时间是准确的，那么20万次的时间就应该是1880毫秒上下，但是我们看到实际情况却是 1034 毫秒，误差达到 42%，所以这种测试方式其实结果很不准确。
