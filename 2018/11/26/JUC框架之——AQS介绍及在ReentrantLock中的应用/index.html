AQS是一个抽象类，用于构建各种同步组件，例如ReentrantLock、CountDownLatch等等。掌握好AQS是深入理解JUC的关键~

1 AQS基础介绍

AQS是一个抽象类，用于构建各种同步组件，例如ReentrantLock、CountDownLatch等等。

1.1 CLH队列

AQS维护一个FIFO同步队列，获取锁失败的线程会放到队列尾进行排队等待唤醒，每次唤醒从head头部开始唤醒。入队需要使用CAS来设置尾节点，出队无需使用CAS，因为出队(release)只有一个线程，不会竞争；入队会有多线程竞争。

1.2 state状态

AQS内部有一个private volatile int state字段，在不同的同步组件中有不同的含义，例如在ReentrantLock中：

  • state == 0 表示没有线程持有锁
  • state == 1 表示已有一个线程持有了锁
  • state > 1 表示有一个线程重复获取了锁(可重入，lock成功多次)，相应的该线程释放锁需要调用对应次数的unlock才能释放掉锁

所谓获取锁成功，其最终体现就是多个线程并发调用AbstractQueuedSynchronizer#compareAndSetState方法，去设置state的状态，由于CAS是原子操作，所以只能有一个线程成功，成功的那个线程即获取到锁。CAS失败的线程会存入CLH队列中被挂起(LockSupport#park(Object))，等待唤醒(LockSupport#unpark(Object))，或者被中断。

1.3 核心方法

1.3.1 获取锁方法

AQS定义了acquire方法，规定了获取锁的方法，而把具体的tryAcquire方法下放到子类来实现，子类可以定义tryAcquire表示获取锁，也可以把它定位为任何有意义的含义来自定义锁。获取锁失败会调用addWaiter来实现入队操作。

1.3.2 释放锁方法

AQS定义了release方法，规定了释放锁的方法，把具体的tryRelease方法下放到子类实现。通过调用unparkSuccessor方法来实现将队列中当前线程节点(head投节点)移除，并唤醒后继节点，无需CSA保证原子性。

2 AQS在ReentrantLock中的应用

2.1 ReentrantLock介绍

ReentrantLock是独占式排它锁，可以通过构造函数来自定义公平性。

ReentrantLock维护一个属性Sync，它继承自 AQS，实现了AQS中部分protected方法，jdk中所有使用AQS的类都采用这种复合模式，而非直接用类继承的方式来使用AQS，所有关于AQS的操作都委托给内部类Sync。

同时有如下两个内部类来区分公平性，二者区别后面会提到。

2.3 ReentrantLock公平性

2.3.1 公平性简述

前面提到，ReentrantLock公平性和非公平的实现，公平性表示按照等待锁的时间先后顺序来获取锁，时间先后顺序是通过CLH队列的FIFO顺序来保障，获取锁的时候会判断当前获取锁的线程是否是CLH队列的head节点线程，不是的话会加入到CLH队列中进行排队。而非公平锁没有这样的限制。

公平锁（Fair）：加锁前检查是否有排队等待的线程，优先排队等待的线程，先排队的线程先得到锁

非公平锁（Nonfair）：加锁时不考虑排队等待问题，直接尝试获取锁，获取不到再到队尾等待

一般来说非公平锁的吞吐量大于公平锁，java中ReentrantLock默认即为非公平锁，内置锁Synchronized也是非公平锁
