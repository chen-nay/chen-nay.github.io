1、CountDownLatch 计数锁存器

1.1 介绍及测试

构造方法：CountDownLatch(int count) 构造方法参数指定了计数的次数。

  • void await() 使当前线程在锁存器倒计数至0之前一直等待，除非线程被中断。

  • boolean await(long timeout, TimeUnit unit) 使当前线程在锁存器倒计数至0之前一直等待，除非线程被中断或超出了指定的等待时间。

  • void countDown() 计数减1。当计数为0，则释放所有等待的线程。

  • long getCount() 返回当前计数。

  • String toString() 返回标识此锁存器及其状态的字符串。

1.2 实现原理

CountDownLatch 的实现原理是通过内部维护Sync类继承AQS，把AQS的状态state当做是count次数。前面的文章提到过AQS设计的巧妙性：内部维护的状态，即能被看做是锁的状态，也能被看做是计数次数等等，tryAcquire方法即能是获取锁(Lock)，也能表示是否可以加入同步队列(CountDownLatch )，关键是看实现方的语义。

CountDownLatch 在初始化时，设置AQS的state值，每调用一次countDown()，state值减1(CAS保证原子性)，减到0时，唤醒AQS条件队列中等待线程(调用了CountDownLatch#await()方法的线程)

注意：

  • 一旦CountDownLatch的计数到0，则无法再将该计数无法被重置。

  • CountDownLatch实例本质与Thread的join方法相同。但join方法仅可以支持当前线程等待一个线程的结束，若需要等待多个线程，则需要逐个线程的调用join方法，非常麻烦。CountDwonLatch可以很方便的实现一个线程等待多个线程。

1.3 使用场景

  • 可以用来让多个线程同时开始运行，对代码做并发测试
  • 多个子线程全部结束后，主线程才继续执行

4、Exchanger 交换器

4.1 介绍及测试

两个线程用来交换数据，两个线程都执行到(不要求同时运行到)exchange方法以后，就相互交换数据，其中有一个线程先运行到同步点(exchange方法),则会阻塞等待另一个线程也运行到同步点，然后开始交换数据。

也即是说Exchanger提供的是一个交换服务，允许原子性的交换两个对象，并且需要成对出现才会成功

  • exchange(V x)：等待另一个线程到达此交换点（除非当前线程被中断），然后将给定的对象传送给该线程，并接收该线程的对象
  • exchange(V x, long timeout, TimeUnit unit)：设置最大等待时长

4.2 使用场景

整体来看Exchanger 还是生产消费模型，主要用在两个线程需要交换对象，假如一个线程实例化某对象的成本很高，另一个线程需要消费这些对象时，使用Exchanger 交换可以较少实例化成本

5 同步工具使用总结

上面4个同步工具个别的内部实现比较复杂(比如Exchanger)，但是对外暴露的API还是比较简单的，上手使用并不难，难点在于业务开发中识别出在什么场景下应该用上同步工具。
